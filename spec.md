# AI MONSTER 実装計画(spec.md)

ゲーム化アンケートLP(ランディングページ:商品やサービスの紹介・誘導用の1枚ページ)のテンプレートシステムの設計書です。
実装前にこの内容でよいかご確認ください。

---

## 1. 全体像

- QRコードからスマホで開く、**縦画面(375px幅基準)専用**のアンケートLP
- アンケートを「横スクロールの簡易RPG(主人公が右へ進む冒険ゲーム)」に変換する
- 主人公が進む → イベント(敵・宝箱など)発生 → アンケート質問が表示 → 回答するとイベント解決して先へ進める
- 全問回答 = ゲームクリア。クリア画面でお礼メッセージと誘導先リンクを表示
- 所要時間2〜3分、質問5〜7問を想定

## 2. engine / config の分離設計(最重要)

### 2.1 考え方

- **engine(エンジン)**:ゲームの仕組みそのもの。全クライアント共通で、1セットだけ存在する
- **config(コンフィグ:設定ファイル)**:クライアントごとの差し替え情報。JSON(テキスト形式の設定ファイル)1つ
- **新規クライアント追加 = configフォルダにJSONを1つ置くだけ**。engineには一切手を入れない

### 2.2 フォルダ構成

```
lp-monster/
├── index.html            ← LP本体(1ファイル。これがengineの入口)
├── engine/
│   ├── engine.js         ← ゲーム進行・画面切替のプログラム
│   ├── questions.js      ← 質問の表示・回答の記録
│   ├── submit.js         ← 回答の送信(GoogleフォームPOST)
│   ├── style.css         ← 共通の見た目(色はconfigの値で上書き)
│   └── themes/
│       └── adventure.js  ← ゲームテーマ「冒険」(初版はこの1つ)
├── configs/
│   ├── cafe.json         ← デモ1:飲食店(カフェ)
│   └── hotel.json        ← デモ2:ホテル
├── spec.md               ← この設計書
└── CLAUDE.md
```

### 2.3 configの切り替え方法

URLの末尾のパラメータ(追加情報)でconfigを指定します。

- カフェ版: `https://○○○.github.io/lp-monster/?config=cafe`
- ホテル版: `https://○○○.github.io/lp-monster/?config=hotel`

QRコードはこのURLごとに発行するだけで、同じengineのまま見た目と質問が切り替わります。
configの指定がない・読み込み失敗の場合は、エラーではなく案内文を表示します。

### 2.4 config.json の項目(スキーマ:設定項目の決まり)

```jsonc
{
  "brand": {
    "name": "店舗・ブランド名",
    "logoUrl": "ロゴ画像のURL(空ならテキスト表示で代替)",
    "mainColor": "#8B5E3C",      // メインカラー(ボタンや見出しの色)
    "accentColor": "#E8A33D"     // アクセントカラー(強調部分の色)
  },
  "theme": "adventure",           // ゲームテーマ名(初版は adventure のみ)
  "title": {
    "heading": "タイトル画面の見出し",
    "subheading": "サブ文言",
    "startLabel": "スタートボタンの文字"
  },
  "questions": [
    {
      "id": "q1",                 // 質問の識別名(送信時の紐づけに使用)
      "type": "single",           // single=単一選択 / multi=複数選択 / text=自由記述
      "text": "質問文",
      "options": ["選択肢A", "選択肢B"],   // type=text のときは不要
      "event": "enemy"            // この質問で発生するイベント(enemy=敵 / chest=宝箱)
    }
  ],
  "clear": {
    "message": "クリア画面のお礼メッセージ",
    "ctaLabel": "誘導ボタンの文字",
    "ctaUrl": "誘導先URL(クーポンページなど)"
  },
  "submit": {
    "googleFormAction": "GoogleフォームのPOST先URL(初版はプレースホルダ=仮の値)",
    "fieldMap": { "q1": "entry.111111" }   // 質問IDとGoogleフォーム項目の対応表
  }
}
```

### 2.5 テーマの拡張構造(将来用)

- テーマ = 「背景・主人公・敵・宝箱などの見た目と演出」のセット。`engine/themes/` に1ファイルずつ追加する方式
- engineは「config の `theme` 名に対応するテーマファイルを読み込む」だけにして、テーマ追加時もengine本体は無修正で済む構造にする
- 初版は **adventure(冒険・ドット絵風)** の1テーマ。絵素材は画像ファイルを使わず、絵文字+CSS(見た目を整える仕組み)で描く
  - 理由:画像の用意・軽量化が不要になり、GitHub Pagesにそのまま置ける制約に合うため

### 2.6 テーマ「premium」(画像素材版)

- 絵文字テーマ(adventure)はそのまま残し、configの `theme` を `"premium"` にすると画像素材で描画されるテーマ
- 確認用に、URLに `&theme=premium` を付けるとconfigを変えずにテーマだけ差し替えてプレビューできる(例:`?config=cafe&theme=premium`)
- 構成:
  - 背景は遠景・中景・近景の3層。層ごとに違う速さで流して**パララックス**(遠くの絵ほどゆっくり動かして奥行きを出す手法)を表現
  - 主人公は歩行コマ(2〜4枚の絵をぱらぱら切り替え)でアニメーション
  - 敵2種・宝箱(閉/開)・ボス・HUD装飾も画像
- 実素材が届くまでは**仮のプレースホルダ画像(SVG)**で動作する。差し替えは「画像を置く+テーマファイルの拡張子設定を1行変える」だけ

#### 必要な素材リスト(AI画像生成用)

| ファイル名 | 用途 | 推奨サイズ(px) | 背景透過 | 形式 |
|---|---|---|---|---|
| bg-far | 遠景(空・山・月など) | 1080×720 | 不要 | PNG納品→WebP変換 |
| bg-mid | 中景(丘・木・建物のシルエット) | 1080×720 | **必要**(絵柄以外は透過) | PNG納品→WebP変換 |
| bg-near | 近景+地面(下端120px程度が地面) | 1080×360 | **必要** | PNG納品→WebP変換 |
| hero-walk-1〜4 | 主人公の歩行コマ(2コマでも可) | 各256×256 | **必要** | PNG納品→WebP変換 |
| enemy-a | 敵キャラ1 | 256×256 | **必要** | PNG納品→WebP変換 |
| enemy-b | 敵キャラ2 | 256×256 | **必要** | PNG納品→WebP変換 |
| chest | 宝箱(閉じた状態) | 256×256 | **必要** | PNG納品→WebP変換 |
| chest-open | 宝箱(開いた状態) | 256×256 | **必要** | PNG納品→WebP変換 |
| boss | ボスキャラ | 320×320 | **必要** | PNG納品→WebP変換 |
| hud-frame | HUD(進行度表示)の装飾枠 | 750×96 | **必要**(半透明可) | PNG納品→WebP変換 |

生成時の注意:
- 背景3層は**左右の端がつながる絵柄**にする(横に繰り返して表示するため)
- 主人公・敵・ボスは**右向き**(主人公は右へ進む構図)
- 歩行コマは**立ち位置と体は同じで、足だけ動かす**(コマ切り替えでガタつかないように)
- 全素材で絵のタッチ(画風)をそろえる

#### 読み込み速度の制約

- 素材合計**2MB以内**を目標。納品されたPNGは組み込み時にWebP(軽量な画像形式)へ変換・圧縮する
- 要確認:変換時の品質設定は実画像を見て調整(目安:背景=品質75前後/キャラ=品質80前後)

## 3. 画面遷移(画面の切り替わり)

```
[1] タイトル画面
     ロゴ/ブランド名・見出し・「スタート」ボタン
        │ タップ
        ▼
[2] ゲーム画面(メイン)
     横スクロールのフィールドを主人公が右へ進む(タップで前進)
     上部に進行度表示:「2/6」の数字+質問ごとのイベントアイコンの列
     (クリア済みのアイコンは✅に変化。あと何問で終わるかが常に見える)
        │ 数タップ進むとイベント発生
        ▼
[3] 質問モーダル(画面に重なる質問ウィンドウ)
     「モンスターが現れた!」等の演出 → 質問表示
     回答するまで閉じられない(=先に進めない)
        │ 回答 → 敵を倒す/宝箱が開く演出 → [2]に戻る
        │ (これを質問の数だけ繰り返し)
        ▼
[4] クリア画面
     クリア演出 → お礼メッセージ → 誘導ボタン(CTA)
     誘導ボタンは画面幅いっぱいの大きいボタンにする(押しやすさ優先)
     このタイミングで回答をまとめて送信
```

## 4. ゲームフロー詳細

### 4.1 操作方式(比較のうえ「タップで進む」方式を採用)

| 方式 | 利点 | 誤動作リスク |
|---|---|---|
| 長押しで歩く | 押している間だけ進む直感的な操作 | **高い**。スマホの長押しはテキスト選択やコンテキストメニュー(長押しで出る端末のメニュー)を誘発しやすい |
| タップで一定距離進む | 1タップ=1歩で誤動作の余地が少ない。連打で進むテンポがゲームらしい | 低い |

- **採用:タップで一定距離進む方式**。1タップで主人公が少し前進し、数タップごとにイベントが発生する
- 補助対策もあわせて実装する:テキスト選択の無効化・長押しメニューの抑制・ダブルタップ拡大の抑制(自由記述の入力欄のみ選択・入力可能のまま残す)
- 迷う要素・失敗する要素は入れない(全員が2〜3分でクリアできることを優先)

### 4.2 イベントと質問の対応

| configの event 値 | 演出 | 想定する使い方 |
|---|---|---|
| `enemy` | モンスター出現 → 回答で撃破 | 通常の質問 |
| `chest` | 宝箱発見 → 回答で開く | 好みや要望を聞く質問 |
| `boss` | ボス出現 → 回答で撃破・そのままクリアへ | 最後の質問(自由記述を想定) |

- イベントの並び順 = configの `questions` の並び順。質問数が5問でも7問でも自動で対応
- 回答内容によるゲーム展開の分岐は**初版では入れない**(全員同じ一本道)

### 4.3 質問タイプ(3種)

| type | 形式 | 回答UI |
|---|---|---|
| `single` | 単一選択 | 選択肢ボタンをタップで即回答 |
| `multi` | 複数選択 | 複数タップ → 「これで攻撃!」ボタンで確定(0個では確定不可) |
| `text` | 自由記述 | 入力欄+確定ボタン(空欄では確定不可。1問のみ想定) |

### 4.4 回答の送信

- 回答は手元(画面内のメモリ)に貯めておき、クリア画面表示のタイミングで**GoogleフォームへPOST送信**(フォーム送信と同じ仕組みでデータを送る方式)
- 初版はプレースホルダURL(仮の送信先)で実装し、送信処理が動くことまでを確認
- **要確認**:Googleフォームへの外部POSTは、ブラウザの仕様上「送信の成否をプログラム側で確認できない」制約があります(送信自体は可能)。実運用時に実フォームで送達テストが必要です

## 5. 技術構成(制約の確認)

- **ビルド不要**:HTML+CSS+JS(素のJavaScript)のみ。CDN(外部ライブラリ配信)も初版は使わず、依存ゼロを目指す
- **GitHub Pagesにそのまま置ける**:全ファイル静的(サーバー処理なし)
- **バックエンドなし**:データ保存はGoogleフォームPOSTのみ
- **スマホ縦画面(375px幅)最優先**:375px幅で全画面・全イベントの表示確認を行ってから完了とする。PC表示は「中央寄せ+左右余白」の最低限対応のみ
- 途中離脱時の回答復元(再開機能)は初版では**対象外**

## 8. 新ゲームモード「runner(群衆ランナー)」【計画・承認待ち】

参考動画(Robloxの群衆ランナー系ゲーム)の遊び心地を、アンケートLPとして簡易再現する新モード。

### 8.1 「モード」という考え方の導入

- これまでの横スクロールRPG風を**モード「quest」**と呼び、新しい縦スクロールランナー風を**モード「runner」**とする
- **モード=遊び方の仕組み**(操作・進行)、**テーマ=見た目**という役割分担
- configに `"mode": "runner"` を1行足すだけで切り替え。**既存のconfig(cafe/hotel)は無修正のまま今まで通り動く**(mode未指定=quest)
- 動作確認用に、URLに `&mode=runner` を付けるプレビューも用意する(テーマプレビューと同じ仕組み)

### 8.2 ファイル構成の変更(engineの整理)

```
engine/
├── engine.js        ← 共通部分(config読込・タイトル/クリア画面・HUD・送信)
├── modes/
│   ├── quest.js     ← 現行のゲーム進行をここに移す(動きは変えない)
│   └── runner.js    ← 新規:群衆ランナーの進行
├── questions.js / submit.js / style.css(共通のまま)
└── themes/(questモード用。runnerは初版CSS描画、将来画像テーマ化可能な構造に)
```

※既存コードの置き場所を整理する作業(リファクタリング)を含むため、questモードが壊れていないかの再確認を必ず行う。

### 8.3 runnerモードの画面とゲームフロー

```
タイトル画面(共通)
   ▼
ランナー画面:
  擬似3D(CSSの遠近表現)の一本道を、仲間の群れが自動で走り続ける
  上部HUD:仲間の数 + 進行度(2/6)
   ▼ 一定時間走るとゲート(質問)が近づいてくる
質問ゲート:
  ・選択肢2〜3個の単一選択 → 道をふさぐ2〜3枚のゲートパネルに
    選択肢を表示。タップしたゲートをくぐる=回答
  ・選択肢4個以上/複数選択/自由記述 → 「大群が道をふさいだ!」
    演出で立ち止まり、従来の回答ウィンドウ(モーダル)を表示
   ▼ 回答すると…
敵集団バトル演出:
  数字付きの敵集団に群れが突撃 → 数字が減って消滅 → 仲間が増える(+演出)
  (必ず勝つ。失敗要素なし)
   ▼ 全問回答後
ボス大群戦 → GAME CLEAR → クリア画面(共通・全幅CTA・回答送信)
```

### 8.4 演出の方針(初版)

- 道・空はCSSのグラデーションと遠近変形で擬似3D表現。**動画のような本物の3Dにはならない**(雰囲気の再現)
- 群れはキャラ絵文字(🏃等)を複数並べて表現。**画面上の表示は最大30体程度**に制限し、それ以上は「×52」のような数字で表す(スマホの性能対策)
- 仲間数の増減はアンケート進行の演出として使う(回答ごとに増える。**減る演出は使わない**:答えて損した気分にさせないため)
- ゲートのパネル色はconfigのメインカラー/アクセントカラーを使用

### 8.5 制約・要確認事項

- 要確認:**実機スマホでの滑らかさ**。群れの体数・アニメーション量は実機確認後に調整する
- 要確認:ゲートの文字数。選択肢の文言が長い場合の折り返し表示は実装時に検証
- 縦長の道はスマホ縦画面と相性が良い構成だが、375px幅での確認を各段階で行う

### 8.6 デモと実装手順(OK後)

1. engineのモード分離(quest.jsの切り出し)→ **既存デモ(cafe/hotel)の回帰確認**
2. runner.js:道と群れの自動走行 → 375px確認
3. 質問ゲート(2〜3択)+モーダル併用(4択以上・複数選択・自由記述)
4. 敵集団バトル演出・ボス戦・クリア画面接続
5. デモconfig `cafe-runner.json`(カフェの質問を流用しmodeだけrunner)を追加し、同一質問で遊び方だけ変わることを証明
